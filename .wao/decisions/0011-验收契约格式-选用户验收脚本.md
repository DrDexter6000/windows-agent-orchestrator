# 0011: 验收契约格式 — 选"用户验收脚本"（acceptance script）

## Context

PRD U13（无对照物场景判对错）提了三选一候选，6+ 处提到但从未收敛：
- A. **行为快照**（behavior snapshot，输入→输出表）
- B. **契约测试**（contract test，类型/形状/不变量，property-based）
- C. **验收脚本**（user-supplied acceptance script，任意逻辑，exit≠0 即不过）

`docs/research/03:212`、`research/05:171`、`01-prd.md:88` 三处都是同一句候选列表，从未取舍。
Niuma 靠 Python 旧版本做对照物（`research/04:88,162`）；WAO 服务全新/半新项目，无对照物，
须由用户/lead 定义契约。

本 ADR 用一个 spike 收敛三选一。

## Spike（2026-06-25，`.dev/p1-spike`）

最小真实任务：写 `add(a,b)` 函数。故意有缺陷实现（字符串拼接，看起来合理）。
三种契约格式各实现一遍，验同一个 buggy + 同一个 correct 实现：

| 候选 | 抓出 buggy？ | 过 correct？ | 表达力 |
|------|-------------|-------------|--------|
| A 行为快照 | ✅（`add(2,3)`=23≠5） | ✅ | 枚举 case，啰嗦但直白 |
| B 契约测试 | ✅（交换律违反+返回 string） | ✅ | 抽象不变量，ad-hoc 任务难写 |
| C 验收脚本 | ✅（assert 抛错） | ✅ | **任意逻辑，完全可表达** |

**关键发现**：A 和 B 都是 C 的退化形式——C 是超集。A = 写死的输入输出（C 的子集）；
B = 不变量断言（也是 C 的子集）。C 用 throw/exit≠0 表达失败，**完全可表达任意验收逻辑**。

## 决策：选 C — 用户验收脚本（acceptance script）

### 理由

1. **超集，零表达力损失**。选 C 不排除 A/B 的用法——用户想写快照就写快照表，想写不变量
   就写断言。选 A 或 B 则人为限制了表达力。

2. **与 WAO 证据链哲学契合**。WAO scorecard 已是"deterministic code, never LLM judgment"
   （SKILL.md §scorecard）。验收脚本 = 一段 deterministic 代码（exit 0/非0），**不引入 LLM 判断**，
   与现有哲学一致。A/B 也符合，但 C 最直接复用现有"命令 exit 门"心智。

3. **与现有 scorecard 机制最小融合**。scorecard 已有 `requireCommands`（命令 exitCode===0）。
   验收脚本就是一条命令——**但它由 lead/用户作为 oracle 提供，由 WAO（或 tester worker）跑，
   而非信任 worker 自己跑自己的产出**。这是与现有机制的关键语义差，但机制层是同一套。

4. **ad-hoc 任务最实用**。lead 临时派活时，写"add(2,3)===5"的断言脚本比抽象不变量（B）
   或维护快照表（A）都快。多数真实验收就是"跑几个例子看对不对"。

### 不选 A/B 的理由

- **A（快照）**：枚举 case 啰嗦；快照维护负担（重生成就是"接受当前行为"，易放过回归）。
  适合成熟 API 的回归测试，不适合 ad-hoc 编排任务。
- **B（契约/property）**：抽象不变量优雅，但**让 lead/user 为每个 ad-hoc 任务想不变量不现实**
  （add 的交换律好想，"feature X 的不变量"难想）。property-based 是高级用法，不该是默认。

### 一个关键语义：谁来跑验收脚本？

**不是 worker 自己跑自己的产出**（那是 `requireCommands`，worker 可伪造/误报）。
验收脚本是 **lead/user 提供的独立 oracle**，由 WAO 在 worker 产出后跑（或派 tester worker 跑，
tester 与 coder 不同源，独立性）。这复用了 WAO 已有的"Auditor/Tester 与 Coder 不同源"原则
（SKILL.md 选型原则）。

## 落地形态（M7-P1 不写代码，只定格式；实现归 P2/P4）

```
# registry/workflow 节点可声明 acceptance：
acceptance: { script: "verify.mjs" }   # 相对 cwd 的验收脚本
# 或 per-run: --acceptance verify.mjs
```

WAO（或 tester worker）在 worker 报完成后跑该脚本：
- exit 0 + stdout → passed，run 可 completed
- exit≠0 或 throw → failed，detail 透传 stderr
- 记 `run.acceptance_checked` 事件进 transcript（与 scorecard.checked 同级）

**与 scorecard 关系**：acceptance 是 scorecard 的一个新 check（`requireAcceptance`），
不是替代。scorecard 验"worker 真干了"（evidence），acceptance 验"干的是对的"（correctness）。
两层。

## 边界 / 不做

- **不做**：内置 LLM 判官（违反证据链哲学；U13 要的是"deterministic"判对错，不是"LLM 觉得对"）。
- **不做**：自动生成验收脚本（那是 P4 编排器的事；P1 只定"用户提供的脚本怎么跑"）。
- **风险**：用户得会写脚本——但 WAO 的用户就是 lead agent（会写代码），可接受。
  非 lead 的终端用户场景留 P4（编排器可能生成验收脚本）。
