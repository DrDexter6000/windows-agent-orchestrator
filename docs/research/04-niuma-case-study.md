# 对标案例：Niuma 多 Agent 无人值守重构（小天fotos 视频）

> 状态：🟡 案例事实已沉淀；对架构的启示已初步提取，仍可迭代。
> 来源：B 站视频「多Agent无人值守跑了4天，14万行代码，怎么编排的？」UP：小天fotos，约 9 分 04 秒。
> 材料形态：用户委托 codex 对视频做内容总结（非逐字稿），可能存在轻度信息损失。
> 日期：2026-06-15 记录。

## 0. 案例一句话

用 Niuma/牛马 的多 Agent 系统把自身从 Python 重构到 TS，连续无人值守跑 4 天 11 小时，
产出 14 万行代码、633 次提交，失败率约 3%。作者认为长任务成功的根因是**图编排 + 结构化 handoff**。

## 1. 实战产量（硬数据）

| 指标 | 数值 |
|------|------|
| 时长 | 6/5 下午 → 6/10 凌晨，4 天 11 小时 |
| 任务运行 | 416 次（374 完成 / 12 失败 / 21 中途被接管） |
| Git 提交 | 633 次 |
| 代码 | 516 文件，141,571 行新增 |
| 测试 | 150 个测试文件，新增代码中 28% 是测试（约 4 万行） |
| Token | 12.9 亿 |
| 失败率 | ~3%，多来自 E2E 环境问题（端口冲突、进程残留、容器没退干净） |
| 硬件 | 8 核 CC150 + 32GB 内存（无并发，压力不大） |

## 2. 核心论点：长任务失败的根因是上下文漂移

作者判断：长任务失败的根因**不是模型不聪明，而是上下文会被反复压缩、遗忘、漂移**。
百万上下文只能推迟问题，不能根治，因为超长任务最终绕不开上下文管理。

**解法**：不要让一个 Agent 背完整长上下文，而是把任务拆成图上的多个小节点，
每个 Agent 在**干净上下文**里完成**单一职责**，再用结构化 handoff 交棒。

> 这一条和 Anthropic 的经验（子代理产出落盘、只回传引用）是同一件事的两面。

## 3. 图编排 / DAG 是核心引擎

- 每个节点是一个 Agent，边是结构化数据。
- 节点之间**不随便聊天，只按预设格式传递结果**。
- 主流水线 6 个节点：分析、编码、跑测试、检查证据、看代码质量、做提交。
- 被调用 **212 次**，几乎所有任务都靠它完成。
- DAG 支持模式：**串行、并行、扇出、汇聚、打回、循环**。
- 关键机制：**数据可以先到，但执行顺序由图上的依赖锁死**。
  - 例：Coder 完成后一次 handoff 同时发给 Tester 和 QualityGate；
  - Tester 立即开工，QualityGate 虽先收到数据，但 `afterTester` 约束使其必须等测试完成。

## 4. 结构化 handoff 是可靠性来源

- 自由对话里 Agent 说"完成了"，系统无法判断它漏了什么。
- 固定格式的交接棒可以被**程序自动检查**：字段缺失、格式不对、证据不完整，都能立刻打回。
- 通信形态：一对一、一对多、多对一。
- 多个调查 Agent 的结果可汇总到一个汇总节点，再生成文档或结论。
- 节点完成交接后由**节点控制器**调度；死循环时发信号给外层编排者接管。

## 5. 总控 Codex 的角色（编排者 vs 执行者分离）

- Codex 是**总控/编排者，不直接 coding**。真正写代码的是 YAML 里定义的 worker 节点。
- Codex 职责三件事：
  1. 把 issue 或语音指令变成结构化任务草稿
  2. 选择 YAML 模板并调用 Manager 创建 run
  3. 轮询节点状态，必要时介入、停止或回滚
- 当前不足：**过度监控，token 消耗偏高**，后续要降低监控频率。

## 6. YAML 是系统行为，不是文档

- 图、节点、交接规则**不是写死在代码里，而是放在 YAML 模板**中。
- 复用：这次 TS 重构同一份 YAML 被调用 212 次，只是每次输入不同。
- 可单独改进：YAML 大致拆成 编排标识 / 图拓扑 / Agent 定义 / 模型 profile / 容器环境 / 安全限位 等层。
  - 模型太贵 → 改 profile
  - 某节点跑偏 → 改它的 system prompt
  - 要加 review → 改节点，不必重写系统
- 作者强调：**配置就是系统行为**。每次跑出问题，把改进写回 YAML，下次所有任务按新标准执行。

## 7. scorecard 审计层（程序审计，不靠 LLM 判断）

- AI 说完成不算，**系统只看证据链**。
- scorecard 检查真实运行记录：节点是否真的跑完、handoff 头格式是否正确、测试命令有没有真实输出、PR 是否关联正确 issue、证据是否闭环。
- 这层是**程序实现的机器审计**，也可按任务复杂度继续定义优化。

## 8. 环境也是编排的一部分

- 每个节点可指定自己的 Docker 镜像和工具链。
- 普通 coding 节点用基础 worker；前端验证节点用 Chrome 镜像（带 Puppeteer）；E2E 容器测试用 Docker 镜像（带 Docker CLI）。
- 配错环境节点起不来；配对环境，每个 Agent 才能在适合的工具链里工作。

## 9. 成功前提与局限

- 关键前提：**有对照物**（Python 旧版本是标准答案，TS 重构完可直接对比行为）。
- 多模块之间有明确协议，E2E 通信测试能验证重构正确性。
- 全新项目更难：AI 还要自己定义"什么算对"。
- **最大问题是慢**：任务和 PR 都串行，每个 PR 无论大小都要过 6 个工位，一个没跑完下一个就要等。

## 10. 后续方向（作者自述）

- 理想做法：**多条流水线并行推进**，每条负责一个模块，互不等待。
- 编排者未来应能**随时设计新编排**，而不是只复用一份 YAML。
- 技术栈收敛到 Electron 桌面面板。

## 11. 对我们架构的启示 🟡（初步提取，待迭代）

### 启示 A：验证了"该确定的确定，该灵活的灵活"

Niuma 的 scorecard + 结构化 handoff + 依赖锁死 = **确定性骨架**；
节点内部怎么写代码 = **留给 LLM 的领域判断**。
这正是我们从 RunMaestro 倒推出来的同一原则，现在有了**第二个独立来源**的正面验证。

### 启示 B：验证了"绝不往 system prompt 灌编排逻辑"

Niuma 的编排逻辑（图拓扑、依赖、handoff 格式）**在 YAML + 引擎里**，不在 agent 脑子里。
agent 只看到"自己的单一职责 task"。这和我们"铁律"一致。
反差：RunMaestro 把编排写成 markdown 灌进 prompt → 崩溃；Niuma 把编排放 YAML → 跑通。

### 启示 C：DAG 依赖模型的一个具体设计点

"数据可以先到，但执行顺序由图上的依赖锁死"——这是一个很重要的区分。
意味着我们的 L3 引擎要区分两个概念：
- **数据依赖**（handoff 的内容何时可得）
- **执行依赖**（节点何时可以开始）
二者解耦，才能支持 Niuma 那种"QualityGate 先收到数据但等 Tester 完成"的精确调度。

### 启示 D：scorecard 是一个值得吸收的独立层

"AI 说完成不算，系统只看证据链"——这是一个**程序审计层**，独立于编排和执行。
在我们的架构里，它对应 L2/L3 之间的一个横切关注点：
- 不信任 agent 自报的"完成"
- 检查 transcript 里的真实事件（命令是否真跑、产出文件是否存在、测试是否真通过）
- 门控判定来自程序检查，不来自 LLM 理解

这条原则强度甚至超过我们 03 草案里的"门控判定是代码判定的"——Niuma 做到了**基于证据链的门控**。

### 启示 E：YAML 即系统行为（配置驱动）

Niuma 把整套编排（拓扑、Agent 定义、模型、环境、安全）放进一份可版本化的 YAML。
- 好处：改进行为不需要改代码，改 YAML 即可，且对所有未来任务生效。
- 这呼应我们"声明式 DAG + 可插拔节点"的方向，但 Niuma 走得更彻底——**配置层是第一公民**。
- 待定 ❓：我们的 YAML schema 要不要也分层到这种粒度（拓扑 / Agent / 模型 / 环境 / 安全）？

### 启示 F：环境/工具链是编排的一部分

Niuma 每个节点可指定 Docker 镜像和工具链。
对我们的影响：
- L1 Backend 接口或 Registry 里，**环境/工具链**可能要成为一等字段（不只 cwd + model）。
- 但我们约束是"无 Docker / 无 WSL 依赖"（AGENTS.md），所以这里要权衡：用 git worktree + 进程级隔离替代容器？还是承认环境隔离是绕不开的？
- 待定 ❓：这是 Niuma 和我们约束的**直接张力点**，需要专门决策。

### 启示 G：总控 Codex 的过度监控是已知坑

Niuma 作者自己承认 Codex 监控过频、token 消耗偏高。
对我们的预警：如果做 LLM 编排器（L3 的可编程策略层），**监控/轮询策略要克制**——
不是越频繁越好。这也呼应我们"可观测性是横切、可空"的设计。

### 启示 H：串行是已知瓶颈，并行编排是下一步

Niuma 的最大痛点是串行慢（每个 PR 过 6 个工位）。
作者自述后续方向就是多流水线并行。
这验证了我们 L2"限并发 + 调度器"和 L3"并行/fan-out"的必要性。

### 启示 I：成功的隐含前提——"有对照物"

Niuma 能跑通的关键前提是 Python 旧版本做对照。
对 life index 这类**全新或半新项目**，"什么算对"本身需要定义。
这提示我们：编排工具可能需要原生支持**验收契约/对照物**作为 scorecard 的输入，
否则全新项目跑起来会缺乏判定标准。

## 12. 与 RunMaestro 案例的对照

| 维度 | RunMaestro（崩溃） | Niuma（跑通） |
|------|-------------------|--------------|
| 编排逻辑位置 | 写进 markdown 灌进 agent prompt | 放在 YAML + 引擎 |
| 状态机/门控 | LLM 自执行，漂移卡死 | 程序审计（scorecard） |
| Agent 上下文 | 全量 playbook + 三源冲突 | 单一职责、干净上下文 |
| handoff | 自由对话，"完成"不可校验 | 结构化、程序可检查 |
| 结果 | 调度链条瘫痪 | 4 天无人值守，3% 失败率 |

**两案对照的结论**：同样是"多 agent 编排"，把确定性逻辑放在哪里，决定了系统是崩溃还是跑通。
这不是模型差异，是**架构差异**。
