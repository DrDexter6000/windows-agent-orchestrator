## 1. 终局产品定义 ✅

> Windows 原生、headless 的 agent runtime 控制平面：统一管理异构本地 agent runtime，
> 提供从单任务到可编程多-agent 工作流的编排，
> 所有状态以事件流持久化，可观测、可回放、可恢复。

**不是**：agent 框架（不实现推理循环）、GUI 看板、云服务。
**是**：runtimes 之上那一层"调度 + 记录 + 编排"的胶水。

## 2. 四层架构

```
┌──────────────────────────────────────────────────────────────┐
│ L4 接口层    CLI client · JSON/JSONL · (可选)本地 HTTP API     │
│             事件订阅(通用化 tail --follow)                   │
├──────────────────────────────────────────────────────────────┤
│ L3 编排层    工作流引擎:内置原语 + 声明式 DAG + 可插拔节点     │
│             节点间传引用(落盘)而非传内容                      │
│             sequential / parallel / fan-out-fan-in / handoff  │
├──────────────────────────────────────────────────────────────┤
│ L2 控制平面  daemon:运行注册表 · 状态机 · 调度器 · 恢复        │
│             生命周期:pending→submitted→running→终态          │
│             优雅关闭 · 限并发 · 优先级队列                     │
├──────────────────────────────────────────────────────────────┤
│ L1 运行时抽象 Backend 接口 · Registry · Transcript            │
│             会话句柄 + 统一消息流(屏蔽 HTTP/stdio/SDK 差异)  │
└──────────────────────────────────────────────────────────────┘
     横切:可观测性(metrics/trace) · 可空性(每层独立可用)
```

### 各层职责与现状

| 层 | 职责 | PoC 现状 | 终局 |
|----|------|---------|------|
| L1 运行时抽象 | Backend 接口 + Transcript | ✅ 已有 opencode-serve backend + JSONL transcript | + 多 runtime + 统一消息流 |
| L2 控制平面 | 状态机、生命周期、恢复 | ❌ 无 daemon，状态在内存 | ✅ 持久 daemon + 确定性状态机 |
| L3 编排层 | 工作流引擎 | 🟡 只有 spawn/run 两种工作流 | ✅ DAG/顺序/并行 + 可插拔节点 |
| L4 接口层 | CLI + 事件订阅 | ✅ 已有 CLI | + 本地 HTTP API + 通用订阅 |

### 关键架构判断 ✅

**Transcript 是控制平面的持久化后端，不是日志。**
当前 retry/status/collect 全靠读 `runs/*.jsonl` 重构状态——这已经是一个"以事件为事实来源"的极简控制平面。终局是把内存里的状态也换成基于 transcript 的确定性状态机，daemon 重启后能无缝接管。

## 3. 三个关键设计决策 🟡

这三个决定"会不会返工"。方向已有共识，细节待 spec 落地。

### 决策 1：Backend 接口抽象到哪一层

不同 runtime 传输差异巨大（OpenCode 是无状态 HTTP + 轮询；Claude/Codex 是进程 + stdout 流；纯 SDK 是无会话 + 返回值）。

**方案**：抽象到"会话 + 消息流"层级，而非 HTTP/stdio 层级——这样加新 runtime 不改上层。

```js
// L1 终局 Backend 接口（伪代码）
class Backend {
  async spawn(agent, task) → RunHandle   // 建会话 + 投递任务
}
interface RunHandle {
  sessionId: string
  events: AsyncIterable<RunEvent>        // 统一消息流，屏蔽轮询/流式差异
  abort(): Promise<void>
}
type RunEvent =
  | { kind: "message", role, parts[] }    // assistant 产出
  | { kind: "tool_use", ... }             // 工具调用
  | { kind: "done", reason, metrics? }    // 完成/失败/超时
  | { kind: "error", error }
```

无论 OpenCode 的"轮询 `/message`"还是 Claude Code 的"读 stdout"，在 Backend 实现里被翻译成**同一个 `AsyncIterable<RunEvent>`**。编排层只面对这个统一形状。

当前 PoC 的 `OpenCodeServeBackend` 已接近这个形状，迁移成本低。

### 决策 2：编排层如何"灵活可调整"

**方案**：定义与引擎分离 + 内置模板 + 可插拔节点。三层灵活性逐级递进：

1. **零代码**：内置模板，`workflow run --template parallel a1 a2`
2. **声明式**：写 `workflow.yaml` 定义 DAG（顺序/并行/条件/handoff）
3. **可编程**：注册自定义节点类型（如"用 LLM 决定分流到哪个 agent"的 router 节点）

**节点间数据传递遵循 Anthropic 经验**：产出落文件系统，节点间只传引用（路径/runId），不传完整内容。transcript 天然就是这个共享黑板。

**DAG 依赖模型（经 Niuma 对照后强化）**：必须区分两类依赖，二者解耦。
- **数据依赖**：handoff 的内容何时可用（A 的产出是 B 的输入）
- **执行依赖**：节点何时可以开始（B 必须**等 A 完成**才能启动，即使数据已到）
- 例：QualityGate 先收到数据，但因 `afterTester` 约束，必须等 Tester 完成才能执行。
  这是 Niuma 的关键设计——**数据可以先到，但执行顺序由图上的依赖锁死**。
- 意味着图引擎要维护两种边，不能只记"数据流向"。

### 决策 3：状态机 + 可恢复性如何与 transcript 结合

显式生命周期（替代当前靠"最后事件 type"的隐式推断）：

```
            ┌──────────┐
            │ pending  │  run 已创建，未提交给 backend
            └────┬─────┘
                 ↓ submit
            ┌──────────┐
            │submitted │  已调 backend.spawn，拿到 sessionId
            └────┬─────┘
                 ↓ 首个 message 事件
            ┌──────────┐     abort      ┌─────────┐
     ┌─────→│ running  │───────────────→│ aborted │
     │      └─┬──┬──┬──┘                 └─────────┘
     │        │  │  │ timeout            ┌─────────┐
     │        │  │  └───────────────────→│timed_out│
     │        │  │ error                 └─────────┘
     │        │  └──────────────────────→┌─────────┐
     │        │                          │ failed  │
     │        ↓ done                     └─────────┘
     │  ┌──────────┐
     └──┤completed │  (retry 时从 completed 回到 pending)
        └──────────┘
```

每次状态转换写一条 transcript 事件（`run.state_change`）。`runs resume <runId>` = 读 transcript → 找无终态的 running run → 重新 attach 或重放。

**确定性铁律**：状态转移条件是代码判定的，**不依赖 LLM 理解**。

**门控的判定输入：证据链，非 LLM 自报（经 Niuma 对照后强化）**：
- **AI 说"完成"不算数**——这是 Niuma 的 scorecard 层的核心思想。
- 门控判定**不信任 agent 自报状态**，而是去查**真实证据**：
  - 测试命令是否真的跑了（查 transcript 里的命令执行记录）
  - 产出文件是否真的存在（查文件系统）
  - handoff 的必填字段是否齐全（查结构化产出）
  - 测试是否真通过（查测试结果记录）
- 这把"门控"从"问 agent 做完没"升级为"**程序审计**"。
- 在我们的架构里，scorecard 是 L2/L3 之间的横切层：编排层触发节点完成，
  但 scorecard **独立检查证据**才允许状态转移到 `completed`。

## 4. 关于"把控制层做薄 / 把灵活性交给 LLM"的讨论收敛 🟡

用户提出的反向思考（汲取 RunMaestro 过厚崩溃的教训）：是否该把控制层做薄、把灵活性交给主控 LLM？

**讨论收敛结论**：不是二选一，是同一件事的两面。

| 主张 | 判断 |
|------|------|
| "别把控制层做厚" | ✅ 完全同意（RunMaestro 验证） |
| "把策略整个交给 LLM，删掉控制层" | ❌ 过头——LLM 编排器只是控制面的**又一个客户端**，和人/bash/YAML 平权 |

**关键洞察**：如果控制面是厚且固执的，它反而**阻碍** LLM 灵活控场（LLM 必须绕开烧死的假设）。
如果控制面是**薄且中立的**（只提供 spawn/wait/collect/abort 这组稳定原语），LLM 才能真正自由发挥。

**实际架构分配**：

```
L4 接口层     薄、稳定
L3 策略层     薄引擎 + 厚策略库 ← LLM 编排器在这里，一等公民，可设为默认
L2 控制平面   薄、确定性、稳定 ← 这层绝不能让 LLM 进来
L1 运行时抽象  薄、稳定
```

**可选性论据**：薄原语设计保留了将来完全 LLM 化的选项（可按工作流逐个调 0%→100%）；纯 LLM 设计提前关闭了确定性的退路（批处理/CI/审计/无人值守/成本敏感场景仍需要确定性）。

## 5. 横切：可观测性 + 独立门控

### 5.1 可观测性

每一层发事件到 transcript，独立 metrics 聚合器读 transcript 产出指标：

| 层 | 事件 | 产出指标 |
|----|------|---------|
| L1 | `backend.request` / `backend.response` | 调用延迟、错误率、token 消耗 |
| L2 | `run.state_change` | 各状态计数、运行时长分布 |
| L3 | `workflow.node.start` / `node.end` | 节点级 trace、DAG 执行图 |

**可空**：不启用 metrics 时，transcript 照常记录，只是不聚合。

### 5.2 scorecard 独立门控层（经 Niuma 对照后新增）

scorecard 是一个**独立于编排和执行的审计层**，挂在 L2/L3 之间：

- **定位**：不信任 agent 自报状态。编排层说"节点完成了"不算数，scorecard 独立检查证据才放行状态转移。
- **检查项**（随实战积累，初版从这些起）：
  - 节点是否真的跑完（查 transcript 事件链是否完整）
  - handoff 头格式是否正确、必填字段是否齐全（查结构化产出）
  - 测试命令有没有真实输出（查命令执行记录 + 测试结果）
  - 产出文件是否真的存在（查文件系统）
  - 证据是否闭环（前驱证据是否支撑当前声明）
- **判定来源**：程序检查，**不来自 LLM 理解**。
- **与状态机的关系**：scorecard 通过 → 允许转移到 `completed`；
  scorecard 不通过 → 阻塞、打回、或转 `failed`，由编排策略决定。
- **为什么独立成层**：如果门控逻辑混进编排引擎，就回到了 RunMaestro 的坑（判定逻辑和调度逻辑纠缠）。
  独立成层才能保证"判定标准"和"调度策略"各自演进、互不污染。

这条原则强度超过原来的"门控是代码判定的"——现在做到了**基于证据链的程序审计**。

## 6. 待定项 ❓

第一轮收敛后，以下决策已固化（见 [05-key-decisions.md](./05-key-decisions.md)），不再列在此处：
- ✅ 隔离方案（worktree + 进程组 + 端口表，Docker 可选）
- ✅ 与 Niuma 的关系（架构参照，不重写）
- ✅ 目标分层（短期可用的薄编排 / 中长期 Niuma 级）
- ✅ daemon 形态（持久 daemon + IPC，但短期可过渡用可重入单进程）
- ✅ runtime 范围（HTTP + 进程式，MCP 不作为 backend）
- ✅ 编排深度（DAG + 可插拔节点 + 配置即行为）

剩余开放问题（不阻塞短期目标，留给 spec 阶段）：
- **验收契约的具体形态** ❓：无对照物场景下"什么算对"怎么定义？
  候选：行为快照、OpenAPI 契约、用户提供的验收脚本。Niuma 靠 Python 旧版本做对照物；
  我们服务全新/半新项目，必须由用户定义契约。
- **YAML schema 字段集** ❓：分层方向已定（拓扑/agent/模型/环境/安全），具体字段待 spec。
- **scorecard 检查规则库** ❓：框架已定（证据链门控），具体检查项随实战积累。
- **IPC 机制** ❓：daemon 阶段再定（命名管道 vs 本地 HTTP）。

## 7. 核心原则清单（汇总，来自 RunMaestro 教训 + Niuma 验证 + 调研）

前 7 条来自 RunMaestro 倒推，被 Niuma 正例验证：
1. ✅ **确定性逻辑绝不外包给 LLM**（状态机、门控、冲突仲裁）
2. ✅ **绝不往 agent 的 system prompt 里灌指令**（铁律，三源冲突的根）
3. ✅ **该确定的确定，该灵活的灵活**（编排逻辑确定性；领域判断留 LLM）
4. ✅ **薄原语 + 厚策略库**（控制面越薄越中立，策略层越自由）
5. ✅ **transcript 是事实来源，不是日志**（持久化、可回放、可恢复）
6. ✅ **节点间传引用不传内容**（Anthropic 经验，防 token 爆炸）
7. ✅ **每层独立可用（可空性）**（L1 能单独跑，L1+L2 有 daemon，L1+L2+L3 有编排）

经 Niuma 对照后新增/强化：
8. ✅ **门控基于证据链，非 LLM 自报**（scorecard 程序审计；AI 说完成不算数）
9. ✅ **数据依赖与执行依赖解耦**（数据可先到，执行顺序由依赖锁死）
10. ✅ **配置即系统行为**（YAML 定义系统行为，改进一次惠及所有未来任务）
11. ✅ **长任务靠节点化 + 干净上下文，不靠堆上下文窗口**（防漂移）
